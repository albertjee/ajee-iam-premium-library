# Failure Recovery Runbook
## Entra Identity Decommissioning Control Plane — Rev3.4

**Audience:** Identity governance consultants and IAM operators  
**Purpose:** Diagnosis and recovery procedures for tool failures  
**Tool:** `Invoke-EntraIdentityDecommissioningControlPlane.ps1`

---

## General Principles

1. Read the exact error message before taking any action. The tool emits structured `[ERROR]` lines that identify the failing gate and the specific check that failed.
2. Never re-run ExecuteRemediation on a partially-failed run without first reviewing the execution log.
3. If a run exits with an error code other than 0, the output folder will still be created. Review its contents before attempting recovery.
4. All Gate A and Gate B failures halt before any Graph write connection is established. No tenant changes occur during gate failures.

---

## Gate A Failures — WhatIf Manifest

Gate A runs first in ExecuteRemediation mode. It validates the WhatIf manifest before any other work begins.

### Failure: WhatIf manifest not found

**Error message:**
```
[ERROR] Gate A validation failed:
  WhatIf manifest file not found: .\out\20260602_143000\...run-manifest.json
```

**Recovery:**
1. Verify the `-WhatIfManifestPath` path is correct and the file exists:
   ```powershell
   Test-Path ".\out\20260602_143000\entra-decommissioning-control-plane-run-manifest-20260602_143000.json"
   ```
2. If the file does not exist, the WhatIfRemediation run may not have completed. Re-run WhatIfRemediation mode with `-GenerateApprovalTemplate` to regenerate the manifest.
3. Ensure the path does not contain spaces without quoting. Always quote paths:
   ```powershell
   -WhatIfManifestPath ".\out\20260602_143000\entra-decommissioning-control-plane-run-manifest-20260602_143000.json"
   ```

### Failure: EngagementId mismatch

**Error message:**
```
[ERROR] Gate A validation failed:
  EngagementId mismatch: manifest='ENG-2026-001', current='ENG-2026-002'
```

**Recovery:**
1. The `-EngagementId` value passed to ExecuteRemediation must exactly match the `EngagementId` recorded in the WhatIf manifest.
2. Check the manifest file's `EngagementId` field:
   ```powershell
   (Get-Content ".\out\20260602_143000\...-run-manifest-20260602_143000.json" -Raw | ConvertFrom-Json).EngagementId
   ```
3. Correct the `-EngagementId` parameter to match exactly (case-sensitive).

### Failure: RunId missing from manifest

**Error message:**
```
[ERROR] Gate A validation failed:
  WhatIf manifest RunId is empty or missing
```

**Recovery:**
1. The manifest was likely generated by an older version of the tool that did not write a `RunId`.
2. Re-run WhatIfRemediation mode to generate a new manifest with a valid RunId:
   ```powershell
   .\Invoke-EntraIdentityDecommissioningControlPlane.ps1 `
       -Mode WhatIfRemediation `
       -TenantId "contoso.onmicrosoft.com" `
       -EngagementId "ENG-2026-001" `
       -ClientName "Contoso Ltd" `
       -Assessor "Albert Jee" `
       -GenerateApprovalTemplate
   ```
3. Send the new WhatIf action plan to the approver for a fresh signature.

## Gate B Failures — Approval Manifest

Gate B runs after Gate A passes. It validates the approval manifest and cryptographic binding.

### Failure: Approval hash mismatch

**Error message:**
```
[ERROR] Gate B validation failed:
  ApprovalEnvelopeHash mismatch: approval manifest hash does not match the WhatIf action plan
```

**Recovery:**
1. The approval manifest was signed against a different version of the WhatIf action plan than the one being used.
2. Do not attempt to manually edit either file. Hashes are computed over the full envelope.
3. Identify which WhatIf action plan the approval was signed against by checking the `WhatIfRunId` in the approval manifest.
4. If the WhatIf action plan was modified after signing (even whitespace changes), the hash will not match. A new approval is required.
5. Rerun WhatIfRemediation to generate a clean action plan, have it approved, and generate a fresh approval manifest.

### Failure: Expired approval

**Error message:**
```
[ERROR] Gate B validation failed:
  Approval has expired. ExpiresUtc=2026-06-01T12:00:00Z, CurrentUtc=2026-06-02T09:00:00Z
```

**Recovery:**
1. Approvals have a fixed validity window. Once expired, the approval manifest cannot be used.
2. Present the original WhatIf action plan to the approver again, or generate a new WhatIf run and request re-approval.
3. When requesting approval, communicate the execution timeline so the approver can set an appropriate expiry.

### Failure: Protected object in approved list

**Error message:**
```
[ERROR] Gate B validation failed:
  Protected object found in ApprovedActions: ObjectId=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

**Recovery:**
1. The approval manifest contains an action targeting a protected object. Protected objects are defined in the tool's protection list and must not be modified.
2. Review the approved actions list and remove the action targeting the protected object.
3. The approval manifest must be regenerated. Contact the approver to issue a corrected approval.

## Execution Failures — During ExecuteRemediation

These failures occur after both gates pass and the Graph write connection is established.

### Partial failures

**Symptom:** Execution completes, but the summary shows `Partial Failed > 0`.

**Recovery:**
1. Review the execution evidence JSON for actions with `Outcome = PartialFailed`:
   ```powershell
   $evidence = Get-Content ".\out\20260602_143000\execution-evidence-20260602_143000.json" -Raw | ConvertFrom-Json
   $evidence.Actions | Where-Object { $_.Outcome -eq 'PartialFailed' } | Select-Object ActionId, ActionType, ErrorDetail
   ```
2. The `ErrorDetail` field will describe which targets succeeded and which failed.
3. For targets that failed, determine whether the failure is transient (Graph throttle, timeout) or permanent (object already deleted, permission denied).
4. Transient failures: re-run with `-ActionId` targeting only the failed actions, after verifying the approval manifest has not expired.
5. Permanent failures: document as accepted risk in the engagement record.

### Graph API errors

**Symptom:** Execution halts with a Graph connection error, or actions fail with Graph API error responses.

**Common causes and recovery:**

| Error | Cause | Recovery |
|---|---|---|
| `401 Unauthorized` | Token expired during execution | Re-authenticate and re-run; write scope token validity is ~1 hour |
| `403 Forbidden` | Missing required scope | Verify the service principal or user account has the required write permissions |
| `429 Too Many Requests` | Graph throttling | Wait 30–60 seconds and re-run with `-ActionId` for the failed subset |
| `404 Not Found` | Target object deleted between WhatIf and execution | Record as resolved finding; no recovery needed |

### Credential removal skips

**Symptom:** `RemoveExpiredApplicationCredential` actions show `OutOfScope` or `Skipped` even though the credential appeared stale in assessment.

**Recovery:**
1. Credential removal skips occur when the credential has been removed by another process between WhatIf and execution, or when the credential's expiry date has changed.
2. This is expected behavior — the tool re-validates staleness at execution time.
3. Review the `ErrorDetail` for the skipped action to confirm the skip reason.
4. Document the skip in the engagement record as a pre-execution change.

## SelfTest Failures

SelfTest runs no Graph queries and requires no tenant connection. Failures here indicate a tool installation or module integrity problem.

### Module load error

**Error message:**
```
[ERROR] SelfTest FAILED:
  Module import failed: Redaction.psm1 — parse error at line 42
```

**Recovery:**
1. Run Gate 1 manually on the failing module:
   ```powershell
   $errors = $null
   [System.Management.Automation.Language.Parser]::ParseFile(
       '.\src\Modules\Redaction.psm1', [ref]$null, [ref]$errors)
   Write-Host "Parse errors: $($errors.Count)"
   ```
2. If parse errors exist, the module file is corrupted or was modified. Restore from the last clean commit:
   ```powershell
   git checkout HEAD -- src/Modules/Redaction.psm1
   ```
3. Re-run SelfTest.

### Schema validation failure

**Error message:**
```
[ERROR] SelfTest FAILED:
  SchemaContracts validation failed: required field 'RunId' missing in execution-evidence schema
```

**Recovery:**
1. A module has been modified in a way that breaks the schema contract defined in `SchemaContracts.psm1`.
2. Run `git diff` to identify what changed:
   ```powershell
   git diff HEAD src/Modules/
   ```
3. Revert unauthorized changes and re-run SelfTest.

### Write cmdlet detected in read-only module

**Error message:**
```
[ERROR] SelfTest FAILED:
  Safety invariant violation: write cmdlet 'Remove-MgGroupMember' detected in Discovery.psm1
```

**Recovery:**
1. A write cmdlet has been introduced into a module that is supposed to be read-only. This is a safety violation.
2. Identify the line in the module and remove or relocate the write cmdlet to the appropriate write-capable module (`Remediation.psm1`).
3. Re-run SelfTest to confirm the violation is resolved.

## Output Export Failures

### Disk space error

**Error message:**
```
[ERROR] Export-DecomAssessmentJson failed: There is not enough space on the disk.
```

**Recovery:**
1. The run output folder requires sufficient disk space. Large tenants with many findings can produce JSON exports of 50–200 MB.
2. Specify a `-OutputPath` on a drive with adequate space.
3. The timestamp subfolder is created at script start — ensure space is available before the run.

### Permission denied on output path

**Error message:**
```
[ERROR] New-Item failed: Access to the path '\\share\assessments\out' is denied.
```

**Recovery:**
1. The output directory must be writable by the current user.
2. Test write access before running:
   ```powershell
   New-Item -ItemType File -Path ".\out\write-test.tmp" -Force | Remove-Item
   ```
3. If using a network share, ensure the share permissions and NTFS permissions both allow write.

## How to Replay from Evidence Bundle

If an execution needs to be reconstructed for audit purposes without re-running the tool:

1. Locate the evidence bundle folder from the run:
   ```
   .\out\20260602_143000\evidence-bundle\
   ```

2. Load the evidence bundle manifest:
   ```powershell
   $bundle = Get-Content ".\out\20260602_143000\evidence-bundle\evidence-bundle-manifest-20260602_143000.json" -Raw | ConvertFrom-Json
   ```

3. Verify file hashes match the recorded SHA-256 values:
   ```powershell
   foreach ($file in $bundle.Files) {
       $hash = (Get-FileHash -Path $file.FullPath -Algorithm SHA256).Hash.ToLower()
       if ($hash -ne $file.Sha256) {
           Write-Warning "Hash mismatch: $($file.FileName)"
       } else {
           Write-Host "[OK] $($file.FileName)" -ForegroundColor Green
       }
   }
   ```

4. Load the run artifacts and pass them to `Invoke-DecomReplayValidation` manually, or re-run the tool with `-GenerateReplayValidation` pointing to the original output path.

## How to Re-run with Clean State

When a run fails mid-execution and the operator wants to start fresh:

1. Do not modify or delete the failed run's output folder — it may contain partial evidence needed for audit.
2. Create a new engagement run with a new output path. If re-using the same EngagementId, ensure the approval manifest has not expired.
3. If the WhatIf manifest RunId changed (because you re-ran WhatIfRemediation), the existing approval is invalid. A new approval is required.

```powershell
# Clean start with explicit new output folder
.\Invoke-EntraIdentityDecommissioningControlPlane.ps1 `
    -Mode Assessment `
    -TenantId "contoso.onmicrosoft.com" `
    -EngagementId "ENG-2026-001-R2" `
    -ClientName "Contoso Ltd" `
    -Assessor "Albert Jee" `
    -OutputPath ".\out\retry"
```

## Error Code Reference

| Exit Code | Meaning |
|---|---|
| `0` | Success |
| `1` | Failure — check console output for `[ERROR]` lines |

The tool does not use exit codes beyond 0 and 1. All diagnostic information is in the console output and in the run folder's output files.
