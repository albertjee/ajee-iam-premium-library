Screen 01 of 03 — The Lure
A LinkedIn ad for Nano-Cents AI targets IT admins who manage Microsoft 365 licensing. The landing page looks professional — pain point messaging, social proof, a familiar Microsoft login button. No red flags. No suspicious domain. The victim clicks Connect with Microsoft 365.
Screen 02 of 03 — Device Code Flow
The victim is redirected to a page showing a device code and the real Microsoft URL login.microsoft.com/device. This is a legitimate Microsoft OAuth device code flow — abused. The spinner says "Waiting for you to sign in." The victim enters the code on the real Microsoft page.
⚠ No malware. No credential harvesting. This is a legitimate Microsoft authentication flow being weaponized.
Screen 03 of 03 — Microsoft Consent Page
The real Microsoft page. Real branding. The warning reads "Do not enter codes from sources you don't trust." The victim ignores it — because every step before this looked completely normal. Consent is granted. Token issued. The attack chain begins.
▶ The warning was there. The victim saw it. They clicked Next anyway.
Screen 01 of 03 — The Call
HELPDESK TICKET — INCOMING
Ticket #HD-20240522-0847 · Priority: HIGH
Caller: Sarah Chen — CFO
Issue: Locked out of M365 account
Location: Singapore (travelling)
Urgency: Board call in 20 minutes
⚠ Caller requesting MFA reset — identity not verified via secondary channel
The caller sounds stressed, professional, and credible. They know Sarah Chen's employee ID, manager name, and last project. The helpdesk agent is under time pressure. Standard procedure requires a callback to the employee's registered mobile — but the caller says they lost their phone. The agent skips the secondary verification step.
Screen 02 of 03 — MFA Reset Approved
Entra ID Admin Center — MFA Reset Log
✓ MFA method removed: sarah.chen@contoso.com

Time: 09:14:32 UTC · Ticket: HD-20240522-0847

MFA reset completed. The attacker logs in immediately from Singapore. Entra ID flags the sign-in as medium risk — anomalous location — but does not block it because no CA policy enforces location-based step-up authentication for Finance users. The session looks like normal Sarah Chen activity.
⚠ The risk flag was raised. No one acted on it. The CA policy gap is the real attack surface.
Screen 03 of 03 — Attacker Inside
Attacker session — logged in as sarah.chen@contoso.com
→ Browsing SharePoint Finance folder
→ Opening Salesforce revenue pipeline
→ Searching Azure AD for accounts with no MFA
→ Found: breakglass-admin@contoso.com · password never rotated
→ Pivot to Global Admin via break-glass account
Session duration so far: 4 minutes 22 seconds
Detection: None
No technical exploit. No malware. The attacker is browsing the tenant as Sarah Chen — everything looks legitimate. The break-glass account is the pivot point: a safety net account with Global Admin rights, no MFA, and a password that was set at provisioning and never rotated.
▶ The real attack surface was the break-glass account — not Sarah Chen's identity.
Screen 01 of 03 — The Offboarding Gap
HR System — Offboarding Ticket

Action required: Revoke all admin access before last day

Marcus Webb was passed over for promotion three months ago. He submitted his resignation on May 8th. HR raised an offboarding ticket — but a misconfigured email routing rule meant IT Security never received it. Marcus retains Global Admin rights for his entire two-week notice period. No one notices.

The backdoor account had been sitting dormant for 16 days. No CA policy ever flagged it as belonging to a departed employee. MFA passed because Marcus enrolled his own authenticator during provisioning. The Salesforce delete looks like an authorised admin action — until someone checks the audit log and finds it was a terminated employee.
▶ The offboarding ticket was closed as complete. The access was never revoked.
